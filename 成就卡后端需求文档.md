# 🎲 TRPG 成就卡片锻造炉 - 线上画廊及后端需求文档

## 1. 项目背景与目标
目前的“TRPG 成就卡片锻造炉”是一个纯前端的单页应用（SPA），数据完全依赖本地存储（IndexedDB）。为了实现**公开展示、用户作品管理和后台管理**功能，项目需要升级为**前后端结合的完整 Web 应用**，并部署到你自己的服务器上。

本项目目标是实现：
1. **公开画廊（大厅）**：所有人可查看已发布（上传）的成就卡。
2. **用户系统**：注册/登录用户可以上传、管理、删除自己的卡片。
3. **管理后台**：管理员对全站数据拥有修改、删除和排序等超级权限。

---

## 2. 前端需求改造建议

针对你的问题：“**需不需要在这个前端文档里做一个专门的展示页面？**”
**答案是：极其需要。** 由于之前的设计是“纯工具向”单页，我们需要在前端扩展出新的路由和页面结构。

### 2.1 新增页面与路由 (Vue Router)
你需要引入 `vue-router`，将原本的单页面重构成多页面结构：
1. **/gallery (线上画廊 - 新增核心页)**：
   * 作为网站的首页（或者单独的广场页）。
   * **功能**：瀑布流或网格形式展示已上传的成就卡（缩略图或基于数据实时渲染）。
   * **分类/排序控件**：时间正序/倒序、按规则（游戏名称）筛选、按热度（如点赞/浏览）、模糊搜索成就名称。
2. **/forge (制卡工具页 - 原有页面)**：
   * 原有功能保留，但新增一个**“上传发布”**按钮（仅登录用户可见或可用）。
3. **/user (个人中心页)**：
   * 注册用户查看自己上传的所有卡片，进行“编辑”、“删除”操作。
4. **/admin (管理后台)**：
   * 仅管理员可见，表格形式管理所有卡片数据与用户。

### 2.2 状态与鉴权
* 引入 Token 鉴权机制（如 JWT），所有需授权的接口请求带上 Token。
* 推荐将当前的 Pinia `useAchievementStore` 增加关于用户登录状态的管理模块 `useUserStore`。

## 3. 后端需求文档 (Backend PRD)

### 3.1 技术栈推荐
考虑到你可能对全栈开发的需求，而且已经有一台云服务器，以下技术栈组合较为合适：
* **后端语言/框架**：
  * *方案 A (Node.js)*：`Express` 或 `NestJS` + `Prisma` (ORM)。推荐！与你的 Vue 前端同为 JS/TS 生态，开发顺手，无需切换思维。
  * *方案 B (Python)*：`FastAPI`。轻量、高性能，如果你熟悉 Python，这是首选。
* **数据库**：`MySQL` (关系型数据库适合这类基础的后台管理系统)。
* **图片存储**：
  * 如果你的服务器带宽和硬盘足够，可以考虑直接存在服务器并在数据库保存路径（例如 `/uploads/cards/xxx.png`）。
  * 如果带宽较小，**强烈建议使用对象存储（如阿里云 OSS、腾讯云 COS）**。成就卡生成的图片如果直接存在数据库会导致数据库瞬间撑爆，而保存在服务器可能导致大厅页面加载缓慢。

### 3.2 数据库结构设计

#### 表1：`User` (用户表)
* `id`: 自增 ID
* `username`: 用户名 (唯一)
* `password_hash`: 加密后的密码 (Bcrypt等)
* `role`: 枚举 `USER` 或 `ADMIN`
* `created_at`: 注册时间

#### 表2：`AchievementCard` (成就卡数据表)
* `id`: 自增 ID
* `user_id`: 关联上传者的 User ID (外键)
* `title`: 成就名称 (用于前台快速检索)
* `game_name`: 游戏名称/规则 (用于前台分类筛选)
* `card_data`: JSON文本，存储卡片的各种内部配置数据 (背景色、样式等，可沿用之前导出的 JSON 结构)。
* `image_url`: 卡片成品图片的 URL 地址 (用于在画廊页面快速渲染展示，而非让前端实时通过 JSON 重绘)。
* `is_public`: 布尔值，是否公开展示在前台。
* `created_at`: 上传发布时间 (用于按时间排序)。

### 3.3 核心 API 接口设计 (RESTful)

#### 【鉴权模块 Auth API】
* `POST /api/auth/register` - 用户注册，创建账号。
* `POST /api/auth/login` - 用户登录，后端验证后返回 JWT Token 供前端后续请求携带。
* `GET /api/auth/me` - 解析 Token，获取当前登录用户信息与权限角色。

#### 【画廊与展示 API (无须登录即可读取)】
* `GET /api/cards/public` - 分页获取公开状态的成就卡列表。
  * **Query 参数**：`page`, `limit`, `sort_by(created_at/title)`, `game_name` (筛选条件)。
* `GET /api/cards/:id` - 获取单张完整的卡片详细 JSON 数据。若玩家需要在此基础上二次更改或浏览详细数据时调用。

#### 【用户制卡模块 (必须带有 User Token)】
* `POST /api/cards` - 制卡并上传发布新成就卡。
  * **接收参数**：含图片的 Base64 或 multipart/form-data 文件，以及 JSON 数据。
* `GET /api/users/me/cards` - 获取登录用户自己上传的所有卡片历史。
* `PUT /api/cards/:id` - 修改自己的某张卡片设置（比如从公开设为私密，或覆盖旧数据）。
* `DELETE /api/cards/:id` - 删除自己的某张卡片。

#### 【管理员模块 (必须带有 Admin Token)】
* `GET /api/admin/cards` - 无视是否公开，拉取全站所有卡片列表，支持高级分页管理。
* `PUT /api/admin/cards/:id` - 管理员直接修改任何卡片的公开状态信息（例如强制下架含有违规内容的卡片）。
* `DELETE /api/admin/cards/:id` - 管理员强制删除违规卡片。
* `DELETE /api/admin/users/:id` - 管理员封禁/删除违规用户。

## 4. 实施发展路线 (Roadmap)
如果由我来协助你完成，可以分为以下几个阶段：

1. **Step 1：选择后端技术栈并搭建基础 API**。先实现最基础的用户注册登录，以及 `AchievementCard` 数据库表的增删改查。
2. **Step 2：前后端联调测试上传图片机制**。成就卡带有自定义图片，需要跑通前端截图 -> 调用后端接口 -> 保存并获取 URL -> 写库的流程。
3. **Step 3：前端 Vue 重构升级**。引入 Vue Router，拆分出“制作页”和“画廊展示页”，为画廊增加网格布局与瀑布流。
4. **Step 4：管理员后台搭建**。在前端新增一个只对管理员角色开放的路由页面，写一个简单的表格管理界面。
